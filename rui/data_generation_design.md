# rui レーン: データ生成設計メモ

> 本ドキュメントは **rui 個人レーンの設計判断** を記録するもの。チーム共有の `要件定義書.md`（main 配下）には記載しない（手法が個人方針のため）。`rui/generate_items.py` / `rui/algorithm_a.py` の設計根拠 SSOT。

## 1. データセット生成規模ロジック

`generate_items.py`（および将来の敵対的データ生成器）が生成する積荷データセットの規模は、現場開示情報を基に以下の論理で決定する。

### 現場開示の事実（いすゞロジスティクス回答 / ケースリスト）

- 最終的な問題規模は **約100コンテナ分**（現状3名で手作業によりバンニングレイアウトを実施中）。
- 1コンテナあたりの搭載実績は **最小8ケース〜最大40ケース** 程度。
- ケース重量は部品搭載後で都度変動するため、client は一律 **1,000〜15,000 kg のランダム** で可と回答。
- 取扱ケースは実運用31種（木箱10種＋スチール21種、ケースリスト参照）。本演習では3種に簡略化して扱う。

### 生成規模の決め方（充填率基準）

- コンテナ搭載可能容積 = 12000 × 2300 × 2400 mm ≒ **66.24 m³**。
- 充填率 **80%** を想定すると、1コンテナに収めるべき積荷の総体積 ≒ 66.24 × 0.8 ≒ **約53 m³**。
- これを満たす積荷量を「1コンテナ ≒ 80ケース相当」の規模目安とし、生成器のスケールパラメータの基準値に据える。
- データセット全体は最終規模（約100コンテナ分）まで、この単位を必要本数ぶん積み上げて構成できるものとする。

## 2. 重量生成（現実的密度モデル / client 公認 divergence）

- client は「重量は都度変動し一概なデータ困難。多少現実と乖離しても 1,000〜15,000 kg ランダムで可」と明言。**[1,000〜15,000] は厳守すべき hard anchor ではなく参考レンジ**。
- 一様 `uniform(1000,15000)`（平均 ≒8,000 kg/個）を全荷物に当てると、コンテナ 24,000 kg 制限に **2〜3 個で到達**し容積を使う前に重量飽和（WEIGHT-BOUND 退化）。3D パッキングのパズルが成立しない。
- そこで `generate_items.py` は **体積比例の密度モデル** `weight = clip(volume_m³ × density, 100, 12000)` を採用。`density` は size_type 別・case 別の現実的帯（kg/m³）から sampling。固定3サイズに単一密度帯は不可（small は ρ≥2117 で非現実、large は ρ>1096 で天井張付き）→ size 別密度必須。これにより全 case が VOLUME-BOUND（容積律速）となり 3D パズルが成立、1コンテナ実効 8〜40 ケースの現実規模に収まる。
- 結果として baseline データ族（`CASE_CONFIGS` 6 case）は意図的に全て VOLUME-BOUND（`sanity_report` で `est_item/container ∈ [8,40]` を assert）。**重量×容積の緊張（WEIGHT-BOUND 寄りの難所）は敵対的生成レーンで density 分布を学習 knob として能動的に作る**（baseline に pre-bake しない）。

## 3. 敵対的生成への適用

上記の体積バジェット（≒0.8×コンテナ容積／本）と密度帯を現実性アンカーとして固定したうえで、（将来31種）サイズ混合比・密度（重量）分布・目的地ID配分を変動させ、レイアウトソルバが最も困る構成（特に weight×volume 緊張）を探索する。これにより「現実的な規模感を保ったまま難しい」インスタンスのみを生成する。文献は `rui/adv_lane/refs/`（AR2L / PAIRED / POET）。

## 4. レイアウト採点（要件定義書 §5 準拠・再掲）

`algorithm_a.py` の `fitness_key` は team 要件定義書 §5.3 の **辞書式順位**に整合: `(失格フラグ, 使用コンテナ数, 重心ズレ平均 |Yg−6000|)` を小さいほど良いとして比較。重み付き総合スコア式・充填率は採点から除外（N の単調関数のため）。重心 ±3000mm はハード制約（超過＝失格）。
