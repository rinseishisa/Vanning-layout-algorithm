# 要件定義書

## 1. プロジェクト目的

本演習の目的は、複数種類の積荷をコンテナへ効率的かつ安全に配置するアルゴリズムを検討し、その設計結果を定量的指標に基づき評価することである。「設計」と「評価」を分離し、データ連携を行うことで、システム設計の基礎であるインターフェース定義と疎結合の概念を理解する。

## 2. システム構成

本システムは、JSON形式のファイルを介して連携する2つのモジュールで構成する。

1. **レイアウト設計モジュール**: 積荷データと制約を入力とし、バンニング配置をJSON出力する。
2. **評価モジュール**: 出力されたJSONを読み込み、物理制約のチェックとスコアリングを行う。

## 3. 物理・制約条件

### 3.1 外部環境（コンテナ・積荷）

- **コンテナ (40ft)**: 12000(L) × 2300(W) × 2400(H) (mm) / 最大積載量 24,000 kg
- **積荷（3種類）**:
  - 小：760(W) × 1130(L) × 550(H) (mm)
  - 中：1490(W) × 2260(L) × 900(H) (mm)
  - 大：2280(W) × 2550(L) × 2355(H) (mm)
  - 各積荷は `generate_items.py` によりランダム生成され、それぞれ1,000kg ~ 15,000kgの範囲で重量と目的地IDが付与される。
- **回転ルール**: 天地固定（Z軸回転禁止）。水平方向（X-Y平面）での90度回転のみ許可する。

### 3.2 配置・運用制約

- **物理制約**: 積荷の重複禁止、コンテナ壁面からのはみ出し禁止、接地制約（宙に浮かせない）。
- **目的地制約**: 1つのコンテナに混載できるのは、同一の目的地IDを持つ積荷のみとする（絶対制約）。
- **重量制限**: 各コンテナの積載合計重量は 24,000 kg を超えてはならない。
- **重心制約**: 各コンテナの合成重量重心 Yg は、コンテナの幾何学的中心 Yc = 6000 から ±3000mm（全長の 25%）以内になければならない。これを超える場合は不安定荷役として失格扱いとする。

### 3.3 データセット生成規模ロジック

`generate_items.py`（および敵対的データ生成器）が生成する積荷データセットの規模は、現場開示情報を基に以下の論理で決定する。

**現場開示の事実（いすゞロジスティクス回答 / ケースリスト）:**

- 最終的な問題規模は **約100コンテナ分**（現状3名で手作業によりバンニングレイアウトを実施中）。
- 1コンテナあたりの搭載実績は **最小8ケース〜最大40ケース** 程度。
- ケース重量は部品搭載後で都度変動するため一律 **1,000〜15,000 kg のランダム** とする。
- 取扱ケースは実運用31種（木箱10種＋スチール21種、ケースリスト参照）。本演習では3種に簡略化して扱う。

**生成規模の決め方（充填率基準）:**

- コンテナ搭載可能容積 = 12000 × 2300 × 2400 mm ≒ **66.24 m³**。
- 充填率 **80%** を想定すると、1コンテナに収めるべき積荷の総体積 ≒ 66.24 × 0.8 ≒ **約53 m³**。
- これを満たす積荷量を「1コンテナ ≒ 80ケース相当」の規模目安とし、生成器のスケールパラメータの基準値に据える。
- データセット全体は最終規模（約100コンテナ分）まで、この単位を必要本数ぶん積み上げて構成できるものとする。

**重量生成（現実的密度モデル / client 公認 divergence）:**

- client は「重量は都度変動し一概なデータ困難。多少現実と乖離しても 1,000〜15,000 kg ランダムで可」と明言。**[1,000〜15,000] は厳守すべき hard anchor ではなく参考レンジ**。
- 一様 `uniform(1000,15000)`（平均 ≒8,000 kg/個）を全荷物に当てると、コンテナ 24,000 kg 制限に **2〜3 個で到達**し容積を使う前に重量飽和（WEIGHT-BOUND 退化）。3D パッキングのパズルが成立しない。
- そこで `generate_items.py` は **体積比例の密度モデル** `weight = clip(volume_m³ × density, 100, 12000)` を採用。`density` は size_type 別・case 別の現実的帯（kg/m³）から sampling。これにより全 case が VOLUME-BOUND（容積律速）となり 3D パズルが成立、1コンテナ実効 8〜40 ケースの現実規模に収まる。
- 結果として baseline データ族は意図的に全て VOLUME-BOUND。**重量×容積の緊張（WEIGHT-BOUND 寄りの難所）は敵対的生成レーンで density 分布を学習 knob として能動的に作る**（baseline に pre-bake しない）。

**敵対的生成への適用:** 上記の体積バジェット（≒0.8×コンテナ容積／本）と密度帯を現実性アンカーとして固定したうえで、31種のサイズ混合比・密度（重量）分布・目的地ID配分を変動させ、レイアウトソルバが最も困る構成（特に weight×volume 緊張）を探索する。これにより「現実的な規模感を保ったまま難しい」インスタンスのみを生成する。

> NOTE: 「8〜40ケース／本」は現場の搭載実績値、「≒80ケース／本」は充填率80%から逆算した生成目標値であり別物。前者は現実性アンカー、後者は生成器のスケール基準として使い分ける。

## 4. データ交換フォーマット（JSON仕様）

座標系は、コンテナの左・手前・下の角を原点 (0,0,0) とし、幅を x、奥行（長手）を y、高さを z とする。

### 4.1 入力データ (`items_input.json`)

`generate_items.py` によって生成される積荷データの仕様。

```json
{
  "dataset_info": {
    "dataset_name": "case_01",
    "seed": 42,
    "item_count": 100
  },
  "items": [
    {
      "item_id": "P001",
      "size_type": "medium",
      "dimensions": {"w": 1490, "l": 2260, "h": 900},
      "weight": 1200.5,
      "destination_id": "DEST_A"
    }
  ]
}
```

### 4.2 出力データ (`layout_output.json`)

レイアウト設計モジュールが出力する配置結果の仕様。

```json
{
  "project_info": {
    "team_name": "Team_Alpha",
    "execution_time_ms": 1250
  },
  "containers": [
    {
      "container_id": 1,
      "destination_id": "DEST_A",
      "total_weight": 18500,
      "items": [
        {
          "item_id": "P001",
          "size_type": "medium",
          "dimensions": {"w": 1490, "l": 2260, "h": 900},
          "position": {"x": 0, "y": 0, "z": 0},
          "weight": 1200,
          "is_rotated": true
        }
      ]
    }
  ]
}
```

※ position は積荷の最小座標（原点に最も近い角）を指す。

## 5. 評価指標とスコアリング

評価チームは以下の手順でレイアウト結果を評価する。**重み付き総合スコア式は採用せず、辞書式順序による順位付けを行う**。

### 5.1 失格判定

レイアウト結果に以下のいずれかの違反が含まれる場合、そのチームは失格となり順位付けの対象から除外する。

- 重複配置（積荷同士が衝突）
- 壁面はみ出し（コンテナ境界外への配置）
- 接地違反（浮いている積荷）
- 目的地混在（1 コンテナに複数目的地）
- 重量オーバー（24,000 kg 超）
- 重心違反（5.2項参照）
- 未配置・重複・未知の積荷（`items_input.json` と `containers[].items` の不一致）

### 5.2 重心バランス（長手方向：Y軸）

ケースサイズの関係上、コンテナ内は1列ではなく横2列でケースが搭載される。そのため現実には前後（Y軸）だけでなく左右（X軸）の重量差も存在するが、本演習では左右方向の重心バランスは考慮外とする。

- **合格**: Yc = 6000 からのズレが ±3000mm（全長の 25%）以内。
- **失格**: ズレが ±3000mm を超える場合。

### 5.3 順位付け（辞書式順序）

合格チーム間の順位は、以下の優先順位で決定する。上位指標で差が付かない場合に限り、下位指標を参照する。

1. **コンテナ数（主指標）**: 使用コンテナ本数が **少ない** チームが上位。
2. **重心ズレ平均（第 1 タイブレーク）**: 全コンテナの重心ズレ `|Yg − Yc|` の平均値が **小さい** チームが上位（走行安定性が高いと評価する）。
3. **処理時間（第 2 タイブレーク）**: `execution_time_ms` が **短い** チームが上位（アルゴリズム効率）。

### 5.4 評価方式の補足
- **平均容積利用率を順位付けから除外する理由**: 総積荷体積 V とコンテナ容積 Vc は固定のため、平均充填率は使用コンテナ数 N の単調関数（`mean(fill) = V / (N × Vc)`）となる。コンテナ数が同点のグループ内では平均充填率も必ず同点となり、タイブレーカーとして機能しないため除外する。
